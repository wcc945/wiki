二. 链路

1作用

一、先纠正一个关键误解

prompt_queue 不是给"LLM 多轮对话"用的——LLM 多轮在主进程里通过会话历史自己管理，根本不需要 prompt_queue。

prompt_queue 是给沙箱容器里那个独立的 agent 进程用的。

主进程里跑的 IronClaw agent 是 LLM 多轮的"主对话"；沙箱容器里跑的 worker 是另一个独立的 agentic 循环实例，主进程通过 prompt_queue 跨进程地驱动它。

二、为什么需要容器级别的 worker

看 src/worker/container.rs 和 src/worker/job.rs，IronClaw 的沙箱 worker 有两种模式：

- ContainerDelegate：完整容器模式，跑完整工具栈（shell、file、apply_patch 等）。
- JobDelegate：轻量 job 模式，跑受限工具集。

它们都跑一个共享的 agentic loop——也就是说，容器里的 worker 自己就是一个 agent，不是简单的"执行单条命令然后退"。

那为什么不让主进程直接调 LLM、非容器工具也走主进程？因为有几类工具必须沙箱隔离：

┌────────────────────────────────────────────────────────────────────────┬──────────────────────────────────────────┐
│                                  工具                                  │              为什么必须沙箱              │
├────────────────────────────────────────────────────────────────────────┼──────────────────────────────────────────┤
│ shell                                                                  │ 能跑任意命令，主进程直接调等于裸奔       │
├────────────────────────────────────────────────────────────────────────┼──────────────────────────────────────────┤
│ file_write / apply_patch / file_read / list_dir（针对 /project/ 路径） │ 写文件系统需要隔离的工作区               │
├────────────────────────────────────────────────────────────────────────┼──────────────────────────────────────────┤
│ 网络出站（HTTP 工具）                                                  │ 需要 allowlist 注入凭据、走 sandbox 代理 │
└────────────────────────────────────────────────────────────────────────┴──────────────────────────────────────────┘

CLAUDE.md 里 "Engine v2 Per-Project Sandbox" 那节明确说：

▎ When SANDBOX_ENABLED=true, engine v2 routes the five filesystem/shell tools for /project/ paths through a per-project Docker container instead of the host filesystem.

所以沙箱是 per-project 的，不是全局的——不同的 project 各跑各的容器。

三、那 prompt_queue 解决的是什么问题

考虑这个用户场景：

用户:  "帮我把 /project 里的 README.md 改一下标题"
[round 1] 主进程 agent 分析 → 决定调 create_job
↓
container_job_manager.create_container(job_1)
↓
容器 worker 启动，自己进入 agentic loop
↓
worker 读 README → file_write 改标题 → 完成
↓
容器 worker 进入 idle，等下一条 prompt

用户:  "顺便把 LICENSE 文件也加一段版权声明"
[round 2] 主进程收到新消息
↓
问题是：这个新消息是发给容器 worker 的（因为 /project/ 工具还在它手里）
还是主进程自己处理？
↓
答案是：发给容器 worker——它才是 /project/ 路径的"主人"
↓
prompt_queue[job_1].send("顺便把 LICENSE 文件也加一段版权声明")
↓
容器 worker 醒过来，继续 round 2 的 agentic loop
↓
完成后继续 idle

用户:  "好，搞定"
[round 3] 容器 worker 完成 → container_job_manager.stop_container(job_1)

关键点： 主进程自己不会去操作 /project/ 下的文件，它只负责"调度和路由"；真正的文件操作 worker 在容器里。所以"用户的第 N 条消息"必须能跨进程回到这个 worker 手里继续跑。

四、和"LLM 多轮"的区别

┌──────────────────┬───────────────────────────────┬───────────────────────────────────────────────────────┐
│       维度       │   主进程 agent 的 LLM 多轮    │                  容器 worker 的多轮                   │
├──────────────────┼───────────────────────────────┼───────────────────────────────────────────────────────┤
│ 状态存在哪       │ 主进程内存 + 会话存储         │ 容器内存                                              │
├──────────────────┼───────────────────────────────┼───────────────────────────────────────────────────────┤
│ 谁驱动 LLM 调用  │ 主进程的 LlmProvider          │ worker 通过 orchestrator 的 LLM proxy（proxy_llm.rs） │
├──────────────────┼───────────────────────────────┼───────────────────────────────────────────────────────┤
│ 历史怎么延续     │ 主进程 session manager 自维护 │ worker 自己 session manager 自维护                    │
├──────────────────┼───────────────────────────────┼───────────────────────────────────────────────────────┤
│ 用户新消息怎么到 │ 直接进主进程消息流            │ 必须经 prompt_queue 跨进程投递给 worker               │
├──────────────────┼───────────────────────────────┼───────────────────────────────────────────────────────┤
│ 何时结束         │ 用户结束会话 / agent 退出     │ worker 完成最后一个工具调用后 idle，等 prompt_queue   │
└──────────────────┴───────────────────────────────┴───────────────────────────────────────────────────────┘

注意 worker 自己也跑 session——也就是说容器里的 worker 内部就是 LLM 多轮的，主进程这一层不参与它的多轮，只负责"投喂新输入"和"回收事件"。

五、再回头看 job_event_tx 的角色

顺着这个图就更清楚了——job_event_tx 是 worker → 主进程的反向通道：

container worker ──(NDJSON/HTTP)──► orchestrator API
│
▼
job_event_tx.broadcast((job_id, user_id, event))
│
▼
main.rs:1024-1042 那个 tokio::spawn
│
▼
dispatch_status_event → gateway SSE → 用户 UI

也就是说：

- prompt_queue = 主进程 → 容器的后续指令通道（用户后续消息）
- job_event_tx = 容器 → 主进程的事件回流通道（工具调用、进度、结果）

两者合起来就是 worker 的"远程控制双向链路"。

六、为什么 container_job_manager 是必须的

回到你最初的疑问——"容器本身不是工具执行吗"——确实，单次工具调用在容器内完成没问题，但 IronClaw 沙箱的设计目标是：

1. 承载完整 agentic loop，而不是单条 shell 命令；
2. 跨多轮用户消息保持状态（worker 内存里的 session、临时文件、cwd）；
3. 生命周期由编排器统一管控（孤儿清理、token 注入、停止/重启）。

container_job_manager 就是这一切的"房东"——它不管工具执行，只管这个"住在 Docker 里的 agent"什么时候搬进来、什么时候搬走、什么时候被强制赶出去。

一句话总结

▎ 沙箱容器里跑的不是"工具执行器"，而是一个独立的 agent 进程。container_job_manager 负责它生灭，prompt_queue 是主进程给它喂后续用户消息的信箱，job_event_tx
▎ 是它给主进程汇报进度的回执。这三个东西合起来才构成完整的"远程代理沙箱"，而不是简单的"工具调用容器化"。

2. 两类沙箱

● 这两个对应 IronClaw 里的两种沙箱执行模式，分别针对不同风险等级的工具。看 src/worker/container.rs 和 src/worker/job.rs 就很清楚。

一、完整容器：ContainerDelegate

对应： full_job routine，或者任何需要 shell / 文件系统写权限的工具调用。

架构：

┌─────────────────────────── host ───────────────────────────┐
│  main ironclaw process                                     │
│    └── orchestrator (HTTP API @ :50051)                    │
│          ▲                                                 │
│          │ NDJSON over `docker exec -i`                    │
│          │                                                 │
│          ▼                                                 │
│  ┌──── Docker container ────────────────────────────────┐  │
│  │  sandbox_daemon binary                                │  │
│  │    └── ContainerDelegate                              │  │
│  │          └── 完整 agentic loop (复用主进程那段代码)    │  │
│  │                ├── LLM (via orchestrator LLM proxy)    │  │
│  │                ├── 全部工具 (shell, file, apply_patch)│  │
│  │                ├── WASM 工具                          │  │
│  │                └── 沙箱网络代理                        │  │
│  │                                                        │  │
│  │  bind mount: ~/.ironclaw/projects/<user>/<project>/    │  │
│  │              → /project/                              │  │
│  └────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘

特点：

- 跑完整的 agentic loop，能用 shell、file_write、apply_patch 之类的高权限工具。
- ~/.ironclaw/projects/<user>/<project>/ 通过 bind mount 挂到容器里的 /project/，sandbox_daemon 通过 NDJSON over docker exec -i 跟主进程对话。
- 网络出站走 src/sandbox/proxy/ 那个代理——它做三件事：
    - 域名 allowlist 过滤；
    - 凭据注入（OAuth bearer 之类）；
    - 流量走 CONNECT 隧道。
- 启动慢、吃资源，但是真正干活的。

二、轻量 Job：JobDelegate

对应： 不需要完整 shell 隔离、但仍然希望异步、不阻塞主对话执行的"长任务"。

架构：

┌──────────────── host ────────────────────┐
│  main ironclaw process                  │
│    └── JobScheduler                      │
│          │                               │
│          ▼                               │
│  ┌──── JobWorker task (in-process) ───┐  │
│  │  JobDelegate                       │  │
│  │    └── 复用同一段 agentic loop      │  │
│  │          ├── LLM                    │  │
│  │          ├── 受限工具集             │  │
│  │          └── 没有 shell/file_write  │  │
│  └────────────────────────────────────┘  │
│                                          │
│  （没有 Docker，没有 /project/ 挂载）      │
└──────────────────────────────────────────┘

特点：

- 跑在主进程内的 tokio task 里，不是独立 Docker 容器。
- 同样跑共享的 agentic loop——也就是说它也是一个独立 agent 实例，有它自己的 LLM 多轮、自己的 session。
- 工具集受限：通常不能调 shell、不能 file_write 到 /project/、不能直接网络出站（要走主进程策略）。
- 启动快、资源轻，适合"发个 webhook 出去然后等回调"这类 IO 密集但风险低的活。

三、为什么共享 agentic loop 但分两个 delegate

src/worker/container.rs 和 src/worker/job.rs 里都有这一段（或者类似的）：

// 两个 delegate 都跑同一份 agentic loop 实现
async fn run_agentic_loop<D: Delegate>(delegate: D, ...) {
// LLM call → tool dispatch → tool execution → next round
}

delegate 模式的好处：

- agentic loop 的核心逻辑（消息流、工具调度、LLM 调用）只写一份。
- delegate 只负责"我是谁"：
    - 我能用什么工具？（delegate.allowed_tools()）
    - 我的文件系统视图是什么？（host 还是 /project/ bind mount）
    - 我的网络出口在哪？（直连 vs sandbox proxy）
    - 我的事件往哪发？（主进程 SSE vs 容器 NDJSON）
    - 我怎么挂掉？（tokio task 退出 vs Docker stop）

所以"完整"和"轻量"的差异完全在 delegate 这一层，loop 本身是共享的。这点 CLAUDE.md 里 "Worker" 那节也提到：

▎ src/worker/container.rs — Container worker runtime (ContainerDelegate + shared agentic loop)
▎ src/worker/job.rs — Background job worker (JobDelegate + shared agentic loop)

四、和前面讲的"prompt_queue + container_job_manager"怎么对上

┌───────────────────────┬──────────────────────────────┬────────────────────────────────────────────────┐
│         组件          │     ContainerDelegate 用     │                 JobDelegate 用                 │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────────────────┤
│ container_job_manager │ ✅  拉起 / 杀掉 / 清理 Docker │ ❌  不需要                                      │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────────────────┤
│ prompt_queue          │ ✅  跨进程投递给容器 worker   │ ⚠️ 同进程直接用 channel，不是这个 prompt_queue │
├───────────────────────┼──────────────────────────────┼────────────────────────────────────────────────┤
│ job_event_tx          │ ✅  容器 → 主进程事件回流     │ ✅  in-process 直接 broadcast                   │
└───────────────────────┴──────────────────────────────┴────────────────────────────────────────────────┘

也就是说——

- JobDelegate 不走 prompt_queue——它和主进程同进程，直接用 tokio::sync::mpsc 之类的 in-process 通道就够了。prompt_queue 是为了跨进程投递这个特定场景才存在的数据结构。
- ContainerDelegate 才用 prompt_queue——因为容器 worker 是另一个 OS 进程。

main.rs:867-880 那段 register_job_tools 里 prompt_queue 只在 config.sandbox.enabled && container_job_manager.is_some() 时才注入，也是这个原因——只有"完整容器"模式才需要跨进程的 prompt_queue。

五、什么时候用哪个

┌──────────────────────────────────────────────────────────┬───────────────────────────────┐
│                           场景                           │            选哪个             │
├──────────────────────────────────────────────────────────┼───────────────────────────────┤
│ 改 /project/ 下的代码、写文件、跑 shell                  │ ContainerDelegate（full_job） │
├──────────────────────────────────────────────────────────┼───────────────────────────────┤
│ 长时间跑一个外部 webhook（比如定时轮询、监听第三方回调） │ JobDelegate                   │
├──────────────────────────────────────────────────────────┼───────────────────────────────┤
│ 网络抓取 + LLM 总结，但不需要写文件                      │ 通常 JobDelegate 就够         │
├──────────────────────────────────────────────────────────┼───────────────────────────────┤
│ 用户说"完整地实现这个 feature，改一堆文件"               │ ContainerDelegate，多轮交互   │
├──────────────────────────────────────────────────────────┼───────────────────────────────┤
│ 后台定期清理任务（heartbeat 类）                         │ JobDelegate                   │
└──────────────────────────────────────────────────────────┴───────────────────────────────┘

一句话总结

▎ ContainerDelegate = 跑在 Docker 里的完整 agent 实例，能用 shell / 文件系统写等高权限工具；JobDelegate = 跑在主进程 tokio task 里的轻量 agent 实例，工具受限但启动快、共享主进程内存。两者共用同一段
▎ agentic loop 代码，只在 delegate 这一层区分能力边界。prompt_queue 和 container_job_manager 是为 ContainerDelegate 这种跨进程场景设计的——JobDelegate 在主进程内自己走 mpsc，根本不需要 prompt_queue
▎ 这个数据结构。

一、先把三层的安全模型摆清楚

关键：每个边界假设的"最坏情况"不同。

二、WASM tool 解决的是什么

WASM tool 的威胁模型非常具体——registry 安装的第三方扩展：

▎ CLAUDE.md → "Skills System":
▎ - Trusted (用户放在 ~/.ironclaw/skills/ 或 workspace skills/, 完整工具权限)
▎ - Installed (registry 安装, 只读工具)

也就是说已安装的扩展默认不可信。WASM 提供的是：

- 内存隔离——线性内存，fuel 计量，OOM 直接 trap，不能访问 host 内存；
- 能力裁剪——host function 必须显式声明（host.rs 里那批函数就是白名单）；
- 网络 allowlist——wasm 工具想出网必须先注册域名；
- 凭据注入——credential_injector 决定哪些 secret 能流进 wasm 工具；
- rate limiter——每个工具独立限流。

所以 WASM 工具的本质是： "我信任你（开发者）写的逻辑，但不信任你能跳出沙箱"。

三、那为什么还需要 ContainerDelegate

WASM 隔离的是单个工具调用。当你做的事情超出"调一个工具"的范畴——比如：

1. LLM 写代码、写文件、再读、再改……多个回合——这不是 WASM 工具的抽象层。WASM 工具只能被动被调一次；
2. agentic loop 本身——主循环怎么调度 LLM、怎么管上下文、怎么跨工具——这些根本不在 WASM 沙箱里，它们跑在主进程（或容器）里；
3. 原生工具（shell、file_write、apply_patch）——这些不是 WASM 工具，是 Rust 直接实现的内建工具。src/tools/builtin/ 里那一堆就是。它们完全没有 WASM 隔离。

看 CLAUDE.md → "Engine v2 Per-Project Sandbox":

▎ engine v2 routes the five filesystem/shell tools (file_read, file_write, list_dir, apply_patch, shell) for /project/ paths through a per-project Docker container instead of the host filesystem.

注意是 for /project/ paths——也就是说：

- 这些内建 Rust 工具本身没有沙箱；
- 沙箱是作用在调用路径上的：路由层（MountBackend vs container）决定实际执行环境；
- 没有 Docker 时降级到 MountBackend，行为不变（CLAUDE.md 里明确写了这个降级路径）。

所以 ContainerDelegate 解决的是： "我信任 agentic loop 的逻辑，但 LLM 在容器里跑一堆 shell/文件工具时，得有个真·OS 级隔离兜底"。

四、为什么三层并存（而不是只用 WASM）

假设只有 WASM 这一层，会出现这些空洞：

┌─────────────────────────────────────┬─────────────────────────────────────────────────────────────┬─────────────────────────────────────┐
│                需求                 │                        WASM 能不能做                        │               谁来做                │
├─────────────────────────────────────┼─────────────────────────────────────────────────────────────┼─────────────────────────────────────┤
│ 第三方扩展工具                      │ ✅  这是 WASM 的设计目标                                     │ WASM tool                           │
├─────────────────────────────────────┼─────────────────────────────────────────────────────────────┼─────────────────────────────────────┤
│ 用户主对话里调 shell "echo hi"      │ ❌  内建 Rust 工具，不走 WASM                                │ 路由层 → host 直接执行 或 container │
├─────────────────────────────────────┼─────────────────────────────────────────────────────────────┼─────────────────────────────────────┤
│ 用户让 LLM 改一个真实项目的一堆文件 │ ❌  不是单次工具调用，需要多轮回合 + 文件系统视图 + 网络出站 │ ContainerDelegate                   │
├─────────────────────────────────────┼─────────────────────────────────────────────────────────────┼─────────────────────────────────────┤

├─────────────────────────────────────┼─────────────────────────────────────────────────────────────┼─────────────────────────────────────┤
│ 用户信任的、本机放的 SKILL.md       │ ✅  Trusted 路径，给完整工具权限                             │ WASM tool with elevated trust       │

三层各有明确的威胁模型，不会互相覆盖：

- WASM tool：隔离不可信代码的"调用颗粒"；
- ContainerDelegate：隔离 LLM 在真文件系统上"做项目"的工作流颗粒；
- JobDelegate：隔离"我不想被它阻塞"的长任务颗粒。

五、WASM tool 在容器里会怎样

这是最反直觉但最关键的一点：

WASM tool 可以跑在 ContainerDelegate 里面。看代码 src/worker/container.rs 里的 delegate 构造：

ContainerDelegate 的工具集 = 内建工具 (受限)  ∪  WASM 工具 (按 trust 配置)

也就是说：

- WASM 工具的"不可信"隔离仍然生效（wasmtime + fuel + 内存）；
- 之外还多一层 Docker 隔离（系统调用、文件系统、网络）；
- 这是纵深防御，不是替代。

CLAUDE.md 里"defense in depth"那条原则在这里就体现了——任何一层失守，下一层还能兜住。

六、对应到 routines / tool 视角

Routines (full_job / scheduled)
│
▼
[ Orchestrator picks a delegate ]
│
├── full_job + SANDBOX_ENABLED=true    → ContainerDelegate (Docker)
│                                          └── tools = builtin (host 文件系统代理) ∪ wasm
│
├── full_job + SANDBOX_ENABLED=false   → JobDelegate (in-process)
│                                          └── tools = builtin (host 直接) ∪ wasm (受限)

WASM 工具始终存在，但内建工具的执行环境按沙箱开关切换 host / container。

七、一句话总结

▎ WASM 隔离的是"单个工具调用"，解决"不可信第三方代码"的威胁模型；ContainerDelegate 隔离的是"整个 agentic loop 在真文件系统上做项目"，解决"LLM 在容器里跑 shell/写文件"的威胁模型；JobDelegate
▎ 隔离的是"我不想被它阻塞的主对话"，解决"长任务"的威胁模型。它们是纵深防御的三层，不是替代——容器里的 WASM 工具仍然受 wasmtime 隔离，WASM
▎ 工具也照样能跑在内建工具的容器化路由后面。每层都有自己的威胁模型假设，单独任何一层都不能覆盖所有场景。

一句话总结

v1 容器里跑的是 ironclaw worker / ironclaw claude-bridge / ironclaw acp-bridge 同一个二进制的长驻子命令进程，通过 HTTP+bearer token 调回 orchestrator。Worker 模式只 inline 执行 8 个 builtin Rust
工具（shell/read/write/list_dir/apply_patch/glob/grep/file_undo），网络完全开放（仅靠 capability drop 限制，没有 domain allowlist —— 那是 v2 per-project sandbox 的计划），文件系统靠 bind mount 隔离到
~/.ironclaw/projects/<job_id>/:/workspace，凭证按 owner-scoped 解密注入到子进程 env，master key 永不进容器。


● 在 IronClaw 这个项目里，容器内没有 wasmtime，也没有 WASM 工具

证据：
- src/worker/container.rs:94 里 WorkerRuntime::new 只调 register_container_tools()，而 register_container_tools 实际只注册 register_dev_tools() 的 8 个 Rust
  内建工具（shell、read_file、write_file、list_dir、apply_patch、glob、grep、file_undo），没有任何 WASM 注册调用。
- src/worker/ 全目录 grep wasm 零命中。
- Dockerfile.worker 镜像里没有 wasmtime 二进制，只有 rust/node/python/gh/claude code CLI。
- Cargo.toml 里 wasmtime 依赖只在顶层 src/ 和 crates/ironclaw_wasm* 里持有，src/worker/ 不引。

WASM 工具实际跑在哪里

WASM 工具只在 host 进程内（带 wasmtime 的那个 IronClaw 主进程）跑。加载入口在 src/tools/wasm/runtime.rs 和 src/tools/wasm/loader.rs，通过 registry.register_wasm(...) 注册。

容器 worker 是另一个独立的 ironclaw binary，不会加载 wasm 模块，也不会持有 wasmtime 实例。两个进程只在 localhost:50051 这条 NDJSON/HTTP 通道上对话。

所以"两层嵌套的凭据注入"在这个仓库里根本不存在

之前讨论的 "WASM tool 跑在容器内 → host function 是 stub → 代理回 host 注入凭据" 这个场景是按纵深防御原则脑补的。当前实现把 WASM 工具和容器工具完全正交化：
- WASM 工具 ↔ host 进程 + credential_injector 注入白名单 header
- 容器 worker ↔ 独立 binary + ProxyLlmProvider（无 LLM 凭据）+ fetch_credentials() 拿到的 child process env

容器 worker 的真实凭据流

容器内接触不到任何架构层的明文凭据：
- LLM API key：ProxyLlmProvider 不持有 key，所有 LLM 请求走 HTTP 到 host 的 orchestrator（:50051），由 orchestrator 持有真实 key。src/worker/proxy_llm.rs:1-4 注释明确写了这一点。
- child process 凭据：worker 通过 IRONCLAW_WORKER_TOKEN 鉴权后调 client.fetch_credentials()（src/worker/container.rs:121-134），从 orchestrator 拿到凭据列表，只注入到 Command::envs()
  启动的子进程（避免污染全局 env），不是 wasm 模块能看到的形式。
- 架构层没有 "容器内的 wasm host function 发请求 → stub 代理回 host" 这条链路，因为容器内没有 wasm。

一句话

WASM 工具和容器 worker 在当前实现里是两个互不重叠的进程：WASM 工具只在 host 进程跑，凭据由 host 上的 credential_injector 注入；容器 worker 是独立 binary，没有 wasmtime，不持有 LLM 凭据，所有 LLM
调用走 orchestrator proxy。两条线不交叉，所以也不存在"凭据穿越两层边界"的路径。

所以容器内部只有8个自建工具？那容器的agent_loop需要外部工具咋搞？比如web_search这些

● 对，容器 worker 默认只能调那 8 个内建工具。需要 web_search 这种外部能力时，容器不会自己去实现或调用这些工具——而是由 host 进程的 agentic loop 在主对话里处理，根本不会通过 worker 的 agentic loop。

关键架构事实：容器 worker 不是"完整 agent"

容器 worker 是为 full_job 这种特定长任务设计的，它跑的是一个受限的 agentic loop：

- ✅  适合：在一个项目目录里改文件、跑 build、跑 test、应用 patch、读 glob 结果、grep 代码
- ❌  不适合：网页搜索、查数据库、调 LLM 用 web 工具、和外部服务对话

为什么？代码里写得很清楚（src/tools/registry.rs:513-521）：

▎ register_orchestrator_tools — Register only orchestrator-domain tools (safe for the main process).
▎ This registers tools that don't touch the filesystem or run shell commands: echo, time, json, http.

而 register_container_tools 是反过来：

▎ Register container-domain tools (filesystem, shell, code). These tools are intended to run inside sandboxed Docker containers.

容器 worker 拿到的工具集是按 domain 切片的——它故意没有 web 类工具。register_container_tools() 注释自己写明（src/tools/registry.rs:527）：

▎ Call this in the worker process, not the orchestrator (unless allow_local_tools = true).

web_search 这类工具实际跑在哪里

在 host 进程的 main agentic loop 里。web_search、memory_search、memory_write、http、web_fetch 这些工具注册在 host 上（通过 components.tools.register_* 那批调用，main.rs 启动时），由 host 的
dispatcher 调度。

典型流：

用户问 "网上 React 19 有什么新特性？"
↓
host 进程 main agent loop
↓ LLM 决定调 web_search
↓
ToolDispatcher::dispatch("web_search", ...)
↓
host 进程内执行 web_search → 拿结果
↓
LLM 看结果，回答用户

整个对话从头到尾在 host 进程内完成，容器 worker 根本不会介入。

什么时候才会触发容器 worker

只有当 host 的 LLM 决定"这个任务是个完整项目级别的活"时——通过 create_job / start_full_job 这种工具。代码主线是 src/tools/builtin/job_tools + src/orchestrator/job_manager.rs：

host agent loop
↓ LLM 决定 "这个活需要容器隔离"
↓ 调 create_job({ ..., tools: ["file_read", "file_write", ...] })
↓
orchestrator
↓ container_job_manager.create_container(job_id)
↓ 容器 worker 启动
↓
worker 跑 agentic loop（只能在那 8 个工具里挑）
↓
结果回传 orchestrator → main.rs:1024 那个 tokio::spawn 投到 SSE
↓
host agent 把 job 结果拼回主对话

所以"web_search + 容器 worker"是什么组合

两种合法组合：

组合 A：web_search 在 host 上做，容器 worker 只做"项目级"子任务

host 主对话
├─ LLM 调 web_search → host 出网 → 拿结果
└─ LLM 决定："够了，现在我去改项目代码"
└─ create_job("把 README 改成最新 API 文档的格式")
└─ 容器 worker: 用 read_file + write_file + apply_patch 改 README
（不会上网，不会调 web_search）

组合 B：job 描述里就预先收集好外部信息，再启动容器

host 主对话
├─ LLM 调 web_search → 拿结果 → 拼成一段 context
└─ LLM 调 create_job({
title: "改 README",
description: "按下面这个 API 文档改：<前序 web_search 摘录>...",
...
})
└─ 容器 worker: description 里已经包含所有外部信息

容器 worker 收到的 job.description 是 host 在 dispatch 时就准备好的"自包含任务说明"。worker 不需要再上网查东西。

这个切片的根本目的，注释里也写到了（src/tools/registry.rs:526）：

以及 src/config/agent.rs 里：

pub allow_local_tools: bool,  // ALLOW_LOCAL_TOOLS env

allow_local_tools = true 时，host 也跑内建 dev 工具；默认 false，dev 工具只能跑在容器里。这是单向隔离——容器只能用内建工具，host 可以额外有 web_search 等。

容器 worker 在工具层面故意比 host 弱——因为：
1. 容器里跑的是 LLM 写的代码（可能不靠谱）
2. 容器网络是受限的（默认可能无外网或走 host proxy）
3. 容器持久的只是 extra_env 里那批 host 显式下发的凭据
4. worker 的 LLM 调用本身经过 ProxyLlmProvider，不能直接调模型

给容器 worker 加 web_search = 打破这个隔离设计。架构上不会让这发生。

一句话总结

▎ 容器 worker 拿到的 8 个工具不是"残缺版"，而是"为那个特定 domain 故意切片的版本"。web_search、memory_*、http、web_fetch 这类外部交互工具根本不会出现在容器 worker 的工具列表里——它们是 host agentic
▎ loop 的工具，由 host 的 dispatcher 调度。需要外部信息的 full_job，host 在 dispatch 时把信息塞进 job.description 里作为任务上下文下发，容器 worker 是个"自包含任务执行器"，不需要再上网。所以**"容器内
▎ agent_loop 需要外部工具"这个场景在当前架构里不会发生**——如果真的需要，那是设计层面的 bug 而不是缺失功能。

5. worker模式

● Worker 模式下容器里跑的是完整 Agent Loop 吗？怎么和主进程（orchestrator）交互？

是的，Worker 模式下容器里跑的是完整 AgenticLoop，不是一个简单的命令执行器。它和 orchestrator 之间是 HTTP + bearer token 的双向 RPC，orchestrator 充当 "大脑"，container worker 充当 "手脚 +
工具执行器"。

直接答案

TokenStore 只存在于 orchestrator 进程（也就是宿主进程、运行 IronClaw 主程序的那个进程）里。容器看到的只是：

2. 它必须自己拿着 token 来敲门
3. 想拿凭据？去 GET /worker/{job_id}/credentials，orchestrator 才会现场去 secrets_store 解密，然后把 {env_var, value} 通过 HTTP 响应返回

容器从 orchestrator 收到的 HTTP 响应体里，从来没有过 TokenStore 这个结构体本身，也没有过它的两个 HashMap 字段。

关键证据

- src/orchestrator/job_manager.rs —— 创建 job 时写入 grants
- src/orchestrator/mod.rs —— 注入到 OrchestratorState

全部都在 src/orchestrator/ 下，没有一处把 TokenStore 的字段值塞到 Docker env、args、bind mount、stdin 或网络数据包传给容器。

一个小但真实的副作用

虽然 TokenStore 的字段值不进容器，但容器收到的明文 secret value 是从 TokenStore 的 grants 列表"派生"出来的——也就是说，如果 grants 列表被污染（比如错误的 secret_name），那解密得到的 value
也会错（甚至解密失败返 403）。

但这跟"字段值传到容器"是两回事。secret_name（grant 的字段之一）也不会出现在响应里——容器只看到自己得到的 env var 叫什么名字，不知道 orchestrator 内部是怎么映射的。
